二、任务1-①：多场景微架构指标采集
================================================================================

【实验目标】
使用 stress-ng 生成5种典型负载，用 perf stat 采集微架构指标

【五类负载】
1. int64      - 整数运算（ALU密集型）
2. matrixprod - 矩阵乘法（浮点/SIMD密集型）
3. read64     - 连续内存读取（访存密集型）
4. rand-set   - 随机内存访问（TLB/Cache Miss密集型）
5. queens     - N皇后问题（分支密集型）

【采集命令】
sudo perf stat -e kmem:mm_page_alloc,kmem:mm_page_free,\
sched:sched_switch,sched:sched_wakeup,\
raw_syscalls:sys_enter,raw_syscalls:sys_exit \
stress-ng --cpu 1 --cpu-method int64 -t 30s

【遇到的困难】
1. 第一个perf stat脚本无法完整运行
   - 原因：命令太长，换行符处理问题
   - 解决：使用反斜杠续行，确保没有多余空格

2. 硬件事件不可用
   - 原因：VMware虚拟化限制
   - 解决：改用tracepoint事件

3. 文件权限问题
   - 问题：sudo生成的文件普通用户无法读取
   - 解决：使用sudo chmod修改权限

【采集结果对比】

| 场景 | 页分配 | 页释放 | 上下文切换 | 系统调用 | 用户态时间 | 内核态时间 |
|------|--------|--------|-----------|---------|-----------|-----------|
| int64 | 825 | 552 | 259 | 506 | 29.98s | 0.003s |
| matrixprod | 1,017 | 548 | 293 | 506 | 29.98s | 0.010s |
| read64 | 1,369 | 1,062 | 322 | 2,032 | 17.94s | 12.05s |
| rand-set | 6,186,227 | 6,185,090 | 272 | 1,042 | 14.92s | 15.16s |
| queens | 824 | 548 | 252 | 506 | 29.98s | 0.005s |

【学到的新知识】
1. tracepoint事件：
   - kmem:mm_page_alloc：内存页分配
   - kmem:mm_page_free：内存页释放
   - sched:sched_switch：进程上下文切换
   - raw_syscalls:sys_enter/exit：系统调用

2. stress-ng负载特征：
   - 计算密集型：用户态时间接近100%，系统调用少
   - 访存密集型：内核态时间占比高，缺页异常多

三、任务1-②：火焰图生成与热点分析
================================================================================

【实验目标】
对两种不同负载生成火焰图，对比分析热点差异

【生成命令】
# 矩阵乘法（计算密集型）
sudo perf record -F 99 -g -- stress-ng --cpu 1 --cpu-method matrixprod -t 30s
sudo perf script > matrixprod.script
./FlameGraph/stackcollapse-perf.pl matrixprod.script | \
./FlameGraph/flamegraph.pl --title="矩阵乘法火焰图" > matrixprod_flame.svg

# 随机访存（访存密集型）
sudo perf record -F 99 -g -- stress-ng --vm 1 --vm-bytes 512M --vm-method rand-set -t 30s
sudo perf script > randmem.script
./FlameGraph/stackcollapse-perf.pl randmem.script | \
./FlameGraph/flamegraph.pl --title="随机访存火焰图" > randmem_flame.svg

【遇到的困难】
1. FlameGraph克隆权限问题
   - 问题：git clone后FlameGraph目录权限不够
   - 解决：使用sudo chown修改所有者

2. perf.data文件权限
   - 问题：sudo perf record生成的文件普通用户无法读取
   - 解决：使用sudo chmod 644修改权限

【火焰图对比分析】

矩阵乘法（计算密集型）：
- 形态：较扁平，热点分散
- 主要函数：stress-ng主函数及计算相关函数
- 内核态函数：极少
- 说明：CPU时间主要在用户态执行计算

随机访存（访存密集型）：
- 形态：高而窄，热点集中（尖塔状）
- 主要函数：exc_page_fault、handle_mm_fault、wp_page_copy
- 内核态函数：大量
- 说明：频繁缺页异常，性能瓶颈在内存子系统

【学到的新知识】
1. 火焰图解读：
   - 尖塔状（高）→ 热点集中，瓶颈明确
   - 扁平状（宽）→ 热点分散，资源利用率均匀

2. 缺页异常处理链：
   exc_page_fault → do_user_addr_fault → handle_mm_fault 
   → handle_pte_fault → do_wp_page → wp_page_copy

3. 系统调用入口：
   entry_SYSCALL_64 → do_syscall_64 → 具体系统调用

四、题目2：Linux 持续 CPU Profiling 工具开发
================================================================================

【实验目标】
开发一个容器化的 7×24 持续 CPU Profiling 工具，支持 `perf` 持续采集、数据轮转、HTTP API 查询和火焰图生成。

【遇到的困难与解决】
1. perf 版本与内核匹配问题
   - 问题：容器启动后 collector 立即退出，日志显示 `WARNING: perf not found for kernel 6.8.0-124`。
   - 排查：容器中 `perf --version` 存在但报错，Dockerfile 安装的是 `linux-tools-5.15.0-130-generic`，宿主机内核为 `6.8.0-124-generic`。
   - 原因：perf 必须与宿主机内核版本一致，容器内核来源于宿主机而不是镜像基础系统。
   - 解决：修改 Dockerfile，安装与宿主机匹配的 `linux-tools-6.8.0-124-generic`。

2. 采集进程意外退出
   - 问题：collector 启动后仅完成一轮就退出，未能持续运行。
   - 排查：`run_collection_loop` 中使用 `set -euo pipefail`，导致任何非零退出码都停止脚本，而 `perf record` 返回非零时脚本直接退出。
   - 解决：在循环中临时 `set +e`，使用 `PIPESTATUS[0]` 获取 `perf record` 的真实退出码，并修正 Bash 算术自增写法为 `iteration=$((iteration + 1))`。

3. 时区导致查询不到文件
   - 问题：测试时记录的北京时间 11:42:55 ~ 11:43:55，但 API 查询返回 0 个文件。
   - 排查：数据目录存在采样文件，文件名时间为 `perf_20260619_034258.data`，说明容器内时区为 UTC。
   - 解决：统一采用 UTC 时间查询，示例 `start=2026-06-19T03:42:55&end=2026-06-19T03:43:55`。

4. 硬件 PMU 事件在容器中不可用
   - 问题：`perf record -e cycles` 在容器中出现权限错误。
   - 解决：设计自动降级机制，优先尝试 `cycles`，失败后使用 `cpu-clock` 软件事件。

```bash
if perf record -e cycles -F 1 -a -o /dev/null -- sleep 0.1 2>/dev/null; then
    PERF_EVENT="cycles"
else
    PERF_EVENT="cpu-clock"
fi
```

【收获】
1. 了解了 perf 与内核版本的强耦合关系，容器内核实际来自宿主机而非镜像。
2. 学会了 Bash 严格模式下 `set -euo pipefail` 的副作用，掌握了使用 `PIPESTATUS` 和临时禁用错误退出的方式。
3. 认识到 Docker 容器默认使用 UTC 时区，查询时间需要统一为 UTC 或显式处理时区。
4. 设计自动降级机制可提升 `perf` 工具在虚拟化/容器环境下的兼容性。

八、任务三：Cache Line 微基准测试
================================================================================

【实验目标】
验证CPU Cache Line大小对数组遍历性能的影响

【实现过程】
1. 编写C程序，测试步长 1,2,4,8,16,32,64,128,256
2. 分配32MB数组，测量各步长的平均访问延迟
3. 用perf stat采集性能数据
4. 对stride=1和stride=64生成火焰图
5. 绘制"步长vs性能"曲线图

【测试结果】

| 步长(字节) | 平均访问时间(ns) | 相对性能 |
|-----------|-----------------|----------|
| 1         | 0.33            | 1.00x    |
| 2         | 0.35            | 1.04x    |
| 4         | 0.41            | 1.24x    |
| 8         | 0.61            | 1.83x    |
| 16        | 0.93            | 2.82x    |
| 32        | 1.55            | 4.68x    |
| 64        | 2.62            | 7.90x    |
| 128       | 3.90            | 11.76x   |
| 256       | 4.66            | 14.06x   |

拐点位置: stride=64 字节（与x86_64 Cache Line大小一致）

【遇到的困难】
1. 使用cat > file << EOF方式创建文件失败
   - 原因：多行输入在终端中提前终止
   - 解决：改用nano编辑器手动创建

2. 程序不支持命令行参数
   - 问题：无法单独测试指定步长，不方便perf采集
   - 解决：修改main函数接受argc/argv，支持命令行参数

3. 火焰图生成时FlameGraph工具路径问题
   - 解决：确认~/FlameGraph/目录存在，使用绝对路径调用

4. 曲线图中文字体显示为方框
   - 原因：matplotlib中文字体未配置
   - 解决：改用英文标签生成图表

【学到的新知识】
1. Cache Line基础知识：
   - x86_64 CPU的Cache Line大小为64字节
   - 步长小于64时，数据在同一个Cache Line内，空间局部性好
   - 步长等于64时，每次访问恰好跨越Cache Line边界
   - 步长大于64时，频繁跨越多个Cache Line，Cache Miss率显著上升

2. 微架构层面的拐点原因：
   - Cache Line是CPU从内存读取数据的最小单位
   - 当访问跨Cache Line时，需要额外加载新的Cache Line
   - 导致L1/L2/L3 Cache Miss增加，访问延迟上升

3. perf工具在Cache Line测试中的应用：
   - 使用perf stat采集page-faults等软件事件
   - 使用perf record生成火焰图，对比不同步长的热点差异

【对话2：编译与运行】
用户："gcc编译报错，undefined reference to `gettimeofday'"

AI："需要包含sys/time.h头文件，并在编译时链接数学库：gcc -O2 -o cache_line_test cache_line_test.c -lm"

用户："运行后stride=1延迟只有0.33ns，这个数据合理吗？"
AI："对于32MB数组，L1 Cache只有32KB，stride=1时顺序访问，空间局部性极好，L1命中率接近100%，0.33ns是合理的。"

【对话3：数据分析】
用户："为什么stride=64出现明显的性能拐点？"

AI详细解释：
- x86_64 CPU的Cache Line大小为64字节
- stride < 64：数据在同一个Cache Line内，只触发一次Cache加载
- stride = 64：每次访问都跨越Cache Line边界，需要加载新的Cache Line
- stride > 64：跳跃更大，预取失效，Cache Miss率进一步上升

用户追问："那stride=128时延迟是stride=64的1.5倍，为什么不是2倍？"
AI："因为硬件预取（Prefetcher）会提前加载相邻的Cache Line，部分缓解了延迟。但预取无法完全消除跨Cache Line的代价。"

【对话4：火焰图生成】
